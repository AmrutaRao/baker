## Why Baker? Explains the reasoning behind Baker and our target audience.
### Modern age API dilemmas: * Where do you implement the *orchestration logic* that involves calling APIs from multiple systems? * What if the logic must run in the *absense of your customer*? * What if the same logic must be exposed to multiple customer channels? * What if your process involves *manual steps in the backoffice*?
## Capabilities you get with Baker? Explains what developers get.
* DSL - domain specific language, achieve more logic with less code * Process - a customer journey, a business process *
* A programming library with a DSL for both Scala and Java * A way to catalogue separate system interactions * A way to describe business process steps to be performed based on the catalogue * A way to keep state and handle functional failures * A way to visually represent the complete process
As a result: * Multiple teams can work together on separate parts of the same orchestration logic * Build once, declare and re-use in many processes *
